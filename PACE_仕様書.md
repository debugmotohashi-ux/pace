# PACE 仕様書

au/UQ外販 月次目標配分・着地見込み管理アプリ
（2026年7月 Ver1.2／PWA＋端末内データ基盤）

---

## 1. 目的と基本思想

月間目標を「会場×曜日区分×条件」ごとの1日あたり件数（レート）に基づいて日次へ配分し、
実績を積みながら着地見込みを可視化、そのまま責任者・スタッフへ共有できるテキストを出す。

基本思想は3つ。

- **数字は全て整数**：現場に出す「◯件/日」は必ず整数。レート×日数＝小計が常に一致する。
- **必達で出す**：現状ペースで足りない場合、許容範囲内でレートを引き上げた「必達値」を共有用の標準とする。
- **手入力は最小**：予定と実績だけ入れれば、配分・展開文は自動生成。

## 3. 画面構成

タブ順：**ホーム / データ入力 / 責任者展開 / スタッフ展開 / 必達モード / レート編集 / 日報取込 / 名称管理 / データ管理 / 取込・月更新 / JSON**

### 3-1. ホーム

- **ダッシュボード（2枚並び）**
  - 現状ペース（ベース）：着地と目標差
  - 必達：着地・目標差・達成/未達バッジ
- **メーター**：実績 ／ 現状計画 ／ 必達上乗せ ／ 開通見込 の4分割＋目標ライン
- **統計**：残数、1日あたり必要数、開通見込 など
- **日別目標テーブル**：計画期間の日ごとの目標
- ヘッダーは safe-area 対応（iOSの時計・電池と重ならない）

### 3-2. データ入力

1. **日次入力**（最上部）
   - 日付／実績件数／見込み件数 → 「この日を保存（実績入力日を基準日に）」
   - **実績を保存すると、その日付が自動で基準日になる**（見込みのみの保存では基準日は動かない）
2. **基本設定**
   - 対象月、基準日、達成期限、月間目標、許容上振れα(%)、開通率(%)、初期累計実績＋その時点(as-of)
3. **一覧ページへのボタン**
   - 「予定の一覧・編集 ▸」／「実績の一覧・編集 ▸」

#### 予定の一覧（サブページ）

- 日付ごとにグループ表示（曜日・祝・土日/平日・件数）
- 各会場の**条件をプルダウンで即変更**、×で削除
- 上部フォームから「この日にこの会場を追加」

#### 実績の一覧（サブページ）

- 日付ごとに **実[ ] 見[ ]** をインライン編集（空欄で削除、×で行削除）
- **● 印**＝累計に反映済みの日（初期累計の翌日〜基準日）
- 基準日より後の実績がある場合、「基準日を◯/◯に進める」ボタンが出る

### 3-3. 責任者展開 / スタッフ展開

生成テキストの表示＋コピー。いずれも**必達値**ベース。

### 3-4. 必達モード（診断用）

ベース計画／必達補正後／一律上振れ%／目標との差。
α上限まで上げても届かない場合は不足件数と対処（期限延長・会場日追加・見込み積み増し）を表示。

### 3-5. レート編集

- 会場ごとに「日種別・条件・レート（件/日）」を編集、行の追加削除
- **並べ替え**：地域▲▼／会場▲▼／条件行▲▼（**この並びが展開文の並びになる**）
- 会場の地域移動（プルダウン）、会場追加・削除
- 店舗：店舗数・1日あたり件数
- **「スケジュールの条件をレートに合わせる」ボタン**（条件名を変えた時の救済。詳細は7章）

### 3-6. 取込・月更新

自分のスケジュール貼り付けフォーマットを読み込み、月次のロールオーバーを行う。

### 3-7. JSON

- 「レート＋予定を書き出す」→ コピー
- 「テキストを読み込む」→ 旧形式の設定・予定を読込
- Ver1.2の全データ保護・端末移行には「データ管理」の暗号化バックアップを使用

## 4. データモデル

```json
{
  "month": "2026-07",
  "goal": 832,
  "alphaPct": 30,
  "basisDate": "2026-06-30",
  "targetDate": "2026-07-26",
  "carryIn": { "asOf": "2026-06-30", "cum": 0 },
  "holidays": ["2026-07-20"],
  "pipeline": { "ratePct": 80 },
  "forecasts": [{ "date": "2026-07-03", "count": 20 }],
  "actuals":   [{ "date": "2026-07-03", "total": 15 }],
  "regions": [
    {
      "name": "札幌", "category": "外販",
      "venues": [
        { "name": "MOP", "slots": [{ "day": "土日", "cond": "", "rate": 30 }] }
      ]
    },
    { "name": "店舗", "category": "店舗", "isStore": true, "storeCount": 6, "storeRate": 6 }
  ],
  "schedule": [{ "date": "2026-07-04", "venue": "MOP", "cond": "" }]
}
```

| フィールド | 意味 |
|---|---|
| `month` | 対象月。月末＝計画の最終候補日 |
| `goal` | 月間目標（件） |
| `alphaPct` | 許容上振れα（既定30%）。必達補正の上限 |
| `basisDate` | 基準日。**この翌日から計画**。実績はこの日までを累計に算入 |
| `targetDate` | 達成期限。月末より前なら、その日までに目標到達する配分 |
| `carryIn` | 初期累計（`asOf` 終了時点までの累計 `cum`） |
| `holidays` | 祝日。**土日として扱う** |
| `pipeline.ratePct` | 開通率(%) |
| `forecasts` | 日次の見込み（**最新日の1件のみ使用**、5-6参照） |
| `actuals` | 日次実績 |
| `regions[].category` | `外販` / `店舗` / その他。展開文の【外販】【店舗】区分に対応 |
| `venues[].slots[]` | `day`（土日/平日）・`cond`（条件文字列）・`rate`（件/日） |
| `schedule[]` | 予定。`date` × `venue` × `cond` |

**並び順の意味**：`regions` の配列順＝地域の並び、`venues` の配列順＝会場の並び、`slots` の配列順＝条件行の並び。展開文・拠点別表示はすべてこの順に従う。

## 5. 計算ロジック

### 5-1. 日種別判定

`holidays` に含まれる日 → **土日**。それ以外は土曜・日曜が**土日**、他は**平日**。

### 5-2. 計画期間

**基準日の翌日 〜 horizonEnd**
`horizonEnd = min(達成期限, 月末)`

### 5-3. レート引き当て

予定1件に対し、`会場` が一致し、`日種別` が一致し、**`条件`が完全一致**する slot の `rate` を使う。
一致するものが無ければ **その予定は計画に乗らない（0件扱い）**。

### 5-4. 累計実績

```
cum = carryIn.cum + Σ actuals.total   （carryIn.asOf < 日付 ≦ basisDate）
```

### 5-5. 必達補正（中核）

```
target = goal − cum − act              ← 残りで積むべき数
base   = Σ round(rate × 1) × 日数      ← 現状ペース（ベース計画）
```

`target > base` の場合、係数 k を 1 から **0.005刻み**で上げ、
`Σ round(rate × k) × 日数 ≧ target` を満たす**最小の k** を採用。

- 上限は `1 + α`（既定 +30%）
- 上限でも届かなければ `capped = true` → 不足件数を正直に表示
- `base ≧ target` なら `k = 1`（下方修正はしない。整数化による多少の上振れは許容）

### 5-6. 整数化と集計

```
rateInt = round(rate × k)          ← 展開文に出る「◯件/日」
subInt  = rateInt × 日数            ← 小計
total   = Σ subInt                 ← 総合計
日別目標 = Σ その日の各会場 round(rate × k)
```

すべて整数で、レート×日数＝小計、日別の合計＝総合計が一致する。

### 5-7. 見込み（開通見込）

```
act = floor(最新日の forecasts.count × 開通率)
```

**合計しない。日付が最も新しい見込み1件のみを「現在の見込み」として使う。**
（例：7/1=10、7/3=20 → 使うのは20。合計30は使わない）

### 5-8. 着地

```
着地（必達）   = cum + total + act
差             = 着地 − goal
着地（現状）   = cum + base + act
```

### 5-9. 店舗の扱い

会場名「店舗」の予定日数 × `storeRate` で1行に集約。`storeCount`（店舗数）は表示のみ。


## 7. 取込（月次ロールオーバー）

**入力形式**（自分のスケジュール貼り付け）

```json
[{ "date": "2026-07-01", "day": "水", "venues": ["ビッグ清田(リスポーン有)", "メガ", "店舗"] }]
```

- 会場トークンは `会場名` または `会場名(条件)`
- 認識する条件語は3つのみ：**外販有 / リスポーン有 / リスポーン+催事有**
- 条件が付かない場合は会場ごとの既定に正規化（`canonCond`）
  - メガ・アークス室蘭 → `外販無`
  - アークス菊水・ビッグ清田 → `リスポーン無`
  - その他 → 空
- スマートクォートは自動補正

**取込時の挙動**

- `month` / `holidays` / `schedule` を差し替え、`targetDate` = 月末、`basisDate` = 初日の前日
- **レートは保持**。新しい会場・条件は **rate 0** で自動追加し、レート編集タブへ誘導
- 「実績もリセット」を選ぶと actuals・goal・carryIn を初期化

**条件表記を変えた時の救済**
レート編集の「スケジュールの条件をレートに合わせる」を押すと、各予定の条件を現在のレートに自動整合。

- 旧→新の対応表（外販あり→外販有、リスポーン様なし→リスポーン無 等）で変換
- その日にその会場の条件が1種類しかなければ自動スナップ
- 判定できないものは日付・会場を列挙（手動で選び直し）

## 8. 運用フロー

**毎日**

1. データ入力 → 日次入力に、その日の実績（と現在の見込み）を保存 → 基準日が自動でその日に
2. 責任者展開／スタッフ展開をコピーして共有

**月初**

1. データ管理で暗号化バックアップを作成
2. 取込・月更新に当月スケジュールを貼り付け → 取込
3. 新会場・新条件があればレート編集でレート入力
4. データ入力で月間目標・達成期限を設定

## 9. 既知の制約・仕様上の注意

1. **条件は完全一致**。表記ゆれがあると計画に乗らず、件数が合わなくなる（救済ボタンあり）
2. **見込みは最新日1件のみ有効**。過去日の見込みは一覧に残るが計算には使われない
3. **データは端末のブラウザ保存のみ**。URL変更・サイトデータ削除・端末故障に備え、暗号化バックアップを定期作成する
4. **必達は全レート一律k倍**。会場ごとの伸びしろの差は反映されない
5. 基準日より後に入れた実績は、基準日を進めるまで累計に反映されない
6. 店舗は1行集約。店舗別の内訳は出せない
7. Service Worker のキャッシュ名は `pace-v12`。更新反映は再デプロイ後にアプリを開き直す運用

## 10. 改善候補（優先度順）

| # | 内容 | 効果 |
|---|---|---|
| 1 | 暗号化バックアップの定期作成リマインド | 端末故障・サイトデータ削除への備え |
| 2 | 条件をIDで管理し、表示名は別テーブル化 | 表記ゆれによる0件化が原理的に起きなくなる |
| 3 | 会場別の必達重み（伸ばせる会場を優先して上げる） | 現場実感に合った必達配分 |
| 4 | 未達時の逆算表示（あと何日延ばせば／どこを何件上げれば届くか） | 判断材料が「不足◯件」から一歩進む |
| 5 | 実績の会場別入力（今は日合計のみ） | 会場別の精度検証・レート更新の根拠になる |
| 6 | 見込みの一覧に「未使用」表示 | 最新1件のみ有効という仕様の誤解を防ぐ |

### 盲点として1つ

**必達モードは月中ほど危険な数字を出す構造**になっている。
残日数が減るほど同じ不足を埋めるためのkが急上昇し、α上限に当たった瞬間に「不足」表示へ切り替わる。
つまり「無理な数字を出し続ける → ある日突然、未達が確定表示される」流れになりやすい。

対策として決めておくべきは、アプリ機能ではなく**運用ルール**。

- kが何%を超えたら「レートを上げる」ではなく「会場日を足す／期限を見直す」に切り替えるか
- 月中のどの時点で目標修正の判断をするか（例：残り1/3の時点でcapped＝会社へ相談）

この閾値を決めておかないと、必達値がそのまま現場への無理な指示になり、数字への信頼が落ちる。

## 11. Ver1.2 日報取込・データ管理

### 11-1. データの優先順位

1. **店舗報告**
   - 唯一の正式実績
   - MNP＋新規をPACEの主実績として使用
   - 来店予約を正式な見込みとして使用
   - 店舗報告を修正した場合は、実績・着地・関連分析を再計算
2. **イベント報告**
   - 会場別獲得数、開通済、見込み、訴求端末を保存
   - 正式実績や正式な見込みへ直接加算しない
   - 会場別の獲得レート計算に使用
3. **イベント累計報告**
   - スタッフ別・商材別の累計獲得と、見込み／開通済の状態を保存
   - 分析・照合・報告漏れ検知専用
   - 実績・レートへ直接加算しない

機種変更、セルアップ、Pixel、新規ID Ptは主実績へ加算しないが、将来分析用KPIとして原文・解析値・累計スナップショットを保存する。Pixelは他実績と重複し得るため、総販売数には加算しない。

### 11-2. 日報取込フロー

1. 報告種別を選択
2. 報告基準日をユーザーが指定
3. 原文を貼り付け
4. 解析
5. 日付、名称、数値、内訳、重複を検査
6. 信頼度と確認画面を表示
7. ユーザー確認後に保存

報告基準日はユーザー指定値を正とし、本文中の日付は照合用とする。両者が違う場合は自動修正せず警告する。

### 11-3. 店舗報告

保存項目：

- 報告店舗
- 報告基準日
- MNP／新規
- 店舗／即日／後日
- 即日・後日の元イベント会場と件数
- 来店予約の対象期間、MNP、新規、合計
- 新規ID Pt、Pixel、機種変更、セルアップの当月累計
- 元報告、解析結果、信頼度、エラー

会場帰属：

- `店舗` → 報告店舗
- `即日` → 括弧内のイベント会場
- `後日` → 括弧内のイベント会場
- 元会場がない場合 → 正式実績には計上するが、会場別レートからは除外

内訳と合計が一致しない場合は保存前に警告する。

### 11-4. イベント報告

保存項目：

- イベント会場
- 報告基準日
- 訴求端末
- 商材
- 開通済
- 見込み
- 総販
- PACE主実績対象かどうか

総販は開通済＋見込みであり、正式実績ではない。機種変更やセルアップを含む場合があるため、MNP＋新規の獲得レート算出時は対象外商材を除外する。

### 11-5. イベント累計報告

イベント累計は加算データではなく、報告日時点のスナップショットとして保存する。

- スタッフ別：商材別の累計獲得
- 商材別：見込み、開通済、累計獲得
- `累計獲得 = 見込み + 開通済`

見込みが開通した場合は見込みを1減らし、開通済を1増やす。累計獲得とスタッフ別累計は変えない。

### 11-6. 正式名称マスタ

管理対象：

- 店舗
- イベント会場
- スタッフ

保存項目：

- 安定ID
- 正式名称
- 別名
- 種別
- 有効／無効
- 登録日
- 最終使用日
- 統合先

判定順：

1. 正式名称の完全一致
2. 登録済み別名の一致
3. 候補が1つなら候補表示
4. 複数候補ならユーザー確認
5. 未登録なら既存名称への割当または新規登録を要求

名称変更や統合後も、報告データは安定IDで関連付ける。使用履歴がある名称の削除は物理削除せず無効化する。

### 11-7. 重複・修正・監査

重複判定キー：

- 店舗報告：報告日＋店舗ID
- イベント報告：報告日＋会場ID
- イベント累計：現在日＋会場ID＋イベント期間

同じキーの報告が再取込された場合：

- 同一内容 → 重複として保存しない
- 差分あり → 更新前／更新後を表示
- 更新理由の入力後、新バージョンを有効化
- 旧バージョンは監査履歴として保持

監査ログには、操作、対象、元報告、解析結果、修正者「本橋」、日時、理由、更新前後を保存する。

### 11-8. レート分析

イベント報告と店舗報告は足し合わせない。

```
獲得レート = イベント報告のMNP＋新規獲得数 ÷ 稼働日数
開通率     = 店舗報告で会場帰属できた開通数 ÷ イベント獲得数
推奨レート = 獲得レート × 開通率
```

- 参照期間：直近60日
- 集計単位：会場×曜日区分×条件
- 3稼働日未満、または対象獲得10件未満は参考値
- 開通率100％超などの異常値は要確認
- 推奨レートは自動反映せず、ユーザー承認後に正式レートへ反映

イベント累計報告はレートへ直接使わず、整合性チェックに使用する。

### 11-9. 暗号化バックアップ・復元

バックアップ対象：

- PACE本体設定、目標、予定、レート、実績、見込み
- 全報告と全バージョン
- 正式名称マスタ
- 監査ログ
- レート分析関連データ
- スキーマバージョン

仕様：

- AES-GCMで暗号化
- PBKDF2-SHA256でパスワードから鍵を導出
- パスワードは端末・バックアップ内のどちらにも保存しない
- ファイル拡張子は `.pacebackup`
- 復元前に内容件数と作成日時を確認
- 復元実行前に現在データを同じパスワードで自動退避
- 復元は追加ではなく、検証後の全置換
- 復元失敗時は現在データを変更しない

### 11-10. エラー・整合性チェック

- 店舗名、会場名、スタッフ名の未登録
- 日付不一致
- 重複報告
- MNP／新規と合計の不一致
- 店舗／即日／後日とカテゴリ合計の不一致
- 元会場内訳と即日／後日件数の不一致
- 来店予約のMNP／新規と合計の不一致
- イベント総販と開通済＋見込みの不一致
- イベント累計とスタッフ別累計の不一致
- 累計減少
- 開通率100％超などの分析異常

### 11-11. Ver1.2の正本

- 正式実績：店舗報告のMNP＋新規
- 正式見込み：最新店舗報告の来店予約
- イベント見込み：分析・照合用の別データ
- 主実績以外のKPI：内部保存し、総数へは加算しない
- 保存場所：端末内IndexedDB
- 端末移行：暗号化バックアップによる手動ブリッジ
